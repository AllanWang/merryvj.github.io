Overview

During Summer 2020, I [remotely] joined Red Hat's UXD organization as an interaction design intern with the App Services team. Here I'll share my process and learnings from working on the Service Registry, an enterprise web application part of Red Hat's Integration suite of middleware products. Red Hat is an open source company – no NDAs on this project!

I was mentored by my manager and 3 other designers. Service Registry = a project in its infancy (...and also my design baby for the summer)


As the sole designer on this project, I conducted the UX design process from discovery to delivering high-fidelity wireframes and prototypes. I worked closely with an engineer, product manager, and user researcher.

Background

The Red Hat Service Registry is a datastore for developer teams to manage evolving versions of "artifacts"

Artifacts can be 2 things: schemas or API designs. These tell developers what the data format being sent and recieved by their applications looks like. As development continues, data needs change. Teams then need a way to keep track of these changes so that they don't write code based on outdated data formats.

Artifacts have:

  • Unlimited versions with multiple versions that can be active at the same time
  • Changing metadata (name, description, labels) that updates with each new version
  • Configurable "rules" that control how compatbile new versions are with old ones
  • External documentation, but only if the artifact is an API design

Source: IBM


It can be used without a UI. But it's nicer with one. It's about to become an important product, so let's get some design magic!

Red Hat had already released an MVP "Tech-Preview" version of the Service Registry, with a basic UI created by engineers. My goal was to be the first light of UX this project had ever seen, creating designs that could be used as frameworks for future iterations of the product.

Design

Artifact details page

This is where a user can view and manage versions of an artifact. It needs to support numerous use cases of uploading a new artifact version, editing metadata, comparing content & documentation between versions, updating artifact rules, and managing state (e.g. "deprecating" a version).

Previewing artifact details with a drawer

Why a preview drawer over an expandable table item?

While the drawer is too narrow to show previews of artifact content, it is much easier to tab through & compare items to see if they match what is being looked for. Additionally, the preview drawer pattern would be consistent with another closely related Red Hat Integration product.


Explorations for table / list items

Table headers or just a list item? Responsiveness?

Options with sortable table headers are desirable for sorting on multiple dimensions (otherwise would have to fit it into the filter toolbar) and even for screen readers. I also asked developers and found that not all of the explorations were technically feasible with the Patternfly design system.

Responsiveness was not a big consideration since this product would not be used on mobile devices. I still did consider screen width and resizing behavior with an open preview drawer.
